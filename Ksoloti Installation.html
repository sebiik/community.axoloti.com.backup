Ksoloti Installation


The install process of Ksoloti is identical to Axoloti BUT you need to install the 1.0.12-2 release and need to do a few extra steps.


Download the release package and runtime package for your OS here:

https://github.com/axoloti/axoloti/releases/tag/1.0.12-2/




The Axoloti community forum is probably down but you can access a backup of the installation guide here:

https://sebiik.github.io/community.axoloti.com.backup/t/quick-start-guide/57.html




Before you connect Ksoloti, the following steps are to be done:


1. Clone or download https://github.com/ksoloti/ksoloti and place it in a location of your choice.




2. Start the patcher, go to preferences and change the "Firmware Dir" to the subfolder "/firmware" of the location where you placed the ksoloti folder.






3. Clone or download https://github.com/ksoloti/ksoloti-factory and place it in a location of your choice.



4. Restart the patcher.



5. In the Patcher preferences, click on the plus sign to add a library. In the resulting library editor window, set the settings as in the screenshot below. Enter the location where you saved the ksoloti-factory folder in the field “Directory”. Add the git URL https://github.com/ksoloti/ksoloti-factory.git in the Remote Path to be able to update the library via the Patcher’s “Sync Libraries” function.


There is probably a simpler way by just selecting git and entering the URL, then Axoloti will download the library automatically, but this whole git business in the stock Patcher is still pretty unintuitive…


That’s it, your Ksoloti should be ready now.


The first time you check the “connect” checkmark in the main window, the patcher might give you a “Firmware CRC mismatch” warning and ask if you want to update the firmware. Click "yes". (Though you may need to click [board > firmware > compile] first)


Do not use the [board > firmware > flash] option because this assumes you want to revert your firmware to the stock Axoloti. The patcher will have the audacity to change the “Firmware Dir” in the preferences back to stock Axoloti, save the preferences, and flash the wrong firmware to your board. In that case you will have to set the “Firmware Dir“ back as described in step 2.


Flashing the stock firmware by error will do no harm, but will simply not work on Ksoloti. Also your Ksoloti may get unresponsive and unable to connect. In that case we’ll need to do do a “DFU upload” – see below.


Usually it is sufficient to restart the patcher and let it offer you to update the board firmware automatically (“Firmware CRC mismatch”), should there be any firmware updates.


Currently there is no easy way to toggle between stock Axoloti boards and Ksoloti boards if you own both. Updating the patcher to recognize both, and at least be able to connect to either Axo or Kso, is planned.




DFU upload “Flash (rescue)”


Disconnect/power down your Ksoloti board, then press and hold S1 (the top of the two switches on the board (or whatever switch you connected to the S1 pin), then power up the board again. The LEDs will stay off which is a good sign. In the Patcher, check via [board > select device...] if the board in DFU mode is recognized. If there is an entry like “STM32 in DFU mode” or similar, click [board > firmware > flash (rescue)] to rescue your board. The firmware upload progress is shown in the console and will take longer than via the regular update: about 1-2 minutes.


Anyway this is why it’s a good idea to connect switches to the S1 and S2 pins in your case design: you can use them as regular switches in your patch but during boot you still have the option to enter DFU mode (via S1), or skip the startup patch if anything goes wrong (via S2).
