I hope it's ok if I post this here.
I'ld like to see to how far I would be able to make objects "on request". I know my own ideas go quite fast, just wondering how far I would come if I'ld be coding other people's ideas..

So, if anyone is in a real need of a module, please, post it below.

Couple of "rules":
1- the idea must be possible (eg. super long fft calculations won't be possible, so don't even ask..)
2- module should be usable for the rest of the community, and as so, could be viewed as "necessary" to be added to the library.
3- if the module is made, PLEASE, delete your request, so to keep things tidy in here.
4- if someone is already working on a module, send a message to the requester (is that the word?), so he can update his request, noting that it's already being taken care off.
5- NO RESPONSES ON MESSAGES! ONLY REQUESTS!!! if you want to respond to someone, send him a personal message!

*update 19/6/2016 -8 new modules are created (I had a really good day coding haha)

toneburst 2016-06-17 11:43:35 UTC #2

I know I'm already in violation of the rules above, but I think it would be good to add that in order to maximise the usefulness of objects to users other than the requester, it would be really good to create help files for each object, or group of related objects, and also if possible to document with comments, both in the patches and in the code itself, where practicable.

Obviously, there's a balance to be struck between documentation and production, but especially with complex objects, having clear examples show exactly how the objects work is enormously useful, and any extra time spent documenting how things work pays for itself because you won't get so many requests for help from confused users.
