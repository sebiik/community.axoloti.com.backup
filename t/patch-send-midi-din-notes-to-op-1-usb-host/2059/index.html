This is my first post, to share a hopefully useful patch for OP-1 owners, and also to get feedback on how to make it better:
midi_DIN_to_USB_Host.axp (5.6 KB)

Background: I wanted to route MIDI DIN notes from my friend's Roland Juno Di to his Teenage Engineering OP-1, using the Axoloti in the middle.

After searching around, I found johannes' patch from an existing thread, "Axoloti as a MIDI USB host?" to route USB Host to DIN. I figured I just needed to swap MIDI_DEVICE_DIN (Roland) and MIDI_DEVICE_USB_HOST (OP-1) in the script, yet this didn't quite work -- it seemed that when routed to the OP-1, the Roland MIDI DIN notes would remain active after releasing the keys.

I had previously tested that the MIDI DIN notes coming in were driving the "electric_piano.axp" example patch, with respective notes turning off when releasing the keys, but was stumped as to why johannes' patch didn't work as I wanted. Then I tried the most basic solution: paste the MIDI script from johannes' patch into the electric piano patch -- now MIDI DIN notes were turning off as expected! After confirming it worked, I tweaked the script to force max note velocity. Admittedly, I didn't clean up the patch before uploading -- it's a work in progress

I'm happy with the result, as the patch works standalone flashed to internal memory, but:

  1. Why does the MIDI script work well in one patch and not the other?
  2. How can I insert the value read from MIDI CC 90 into the MIDI script?

Thanks!

My second question, how do I use a midiCC output value from a midi/in/cc object into a MIDI script? Global variables?

no its far easier just to create a patch/object and put your code from the midi script in the midi handler.
then you can add inlets to the object (to get input from the midi cc) , that object can then have variables which you can access (define in local data)

one thing you have to understand though is that midi input is NOT executed the k-rate cycle, this is why you will see in the midi object (check any midi/in object) data is saved in a variable in the object, and then moved to the outlet in the k-rate code.

(this may not make a big difference to you code it, but might avoid some confusion )
