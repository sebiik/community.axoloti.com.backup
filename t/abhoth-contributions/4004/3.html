Abhoth 2017-12-10 15:18:30 UTC #2

abhoth/wave/polyplay

Description:
A variant on factory/wave/play intended for polyphonic use. Streams audio from sd-card. With a sample rate control for pitch.

The rate control/linear interpolation is a bit rough, but in my opinion good enough for the Mellotron patch I wanted it for. If smoother pitch control or higher audio fidelity is needed it needs more work. In addition to being less efficient than the facory/wave/play object it also uses more memory (because of a third buffer for reading from sd-card, which then applies the resampling to the pingpong buffers).

Abhoth 2017-12-10 15:26:33 UTC #3

abhoth/AxoBlock/Mellotron

Description:
6 voice Mellotron with 10 selectable tapes.

https://www.instagram.com/p/BchpjWGj-Qb/?taken-by=abhoth_

This happens to be an MPE patch simply because the midi controller I use for it sends MPE midi. Expression isn't actually used, so it would be easy to convert it into a "normal" polyphonic midi patch.

The samples used for this patch are a free sample pack from Sonic Bloom converted to 48kHz .raw. Individual files renamed with midi note number instead of "C#3" etc. Redistributed with permission from Sonic Bloom. The original, free sample library can be found at http://sonicbloom.net/en/

The sample folders in the archive should go to the root folder of the sd-card. This patch is made specifically for my Axoloti project. Some parameters are in my case modulated by physical knobs/push buttons. So open the cut-off frequency of the filter completely and turn up the volume on the audio out for a non-modulated, "normal" starting point.

Abhoth 2017-12-12 07:47:45 UTC #7

It is an MPE patch, so midi channel 1 isn't going to work (that is the master channel, the rest are for notes).

The noise issues doesn't sound good, though! I don't have any clicks, but when I first tried doing a Mellotron patch using the factory/wave/play object I had ugly clicks whenever a sample was playing and a new one was started. What I believe was happening was that when the sd-card was busy seeking the start of the sample data, the thread responsible for reading from the sd card took so long that audio rate cycles were skipped. Hence the clicks. So I bought the fastest sd-card I could find. When that didn't appear to help, I made the "polyplay" object with lower thread priority. Which fixed it. But I didn't go back to try the old, slow card. So I don't actually know if a sd-card with poor seek time will cause problems. It seems likely, I guess, but it could be something else that is wrong and just happens to work out on my specific card.

I haven't noticed any distortion, but to rule that out it is probably worth it to attenuate the audio from the polyplay object, just in case.
